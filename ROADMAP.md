# HENU-Kit Roadmap

> 当前阶段：从“项目合集”升级为“统一产品入口”  
> 原则：规则先行、复用现有产品、一次只解决一个核心任务。

## 1. 产品目标

HENU-Kit 面向河南大学学生，提供简洁、统一的校园工具入口。它不是新的大一统业务系统，也不把所有项目代码合并到同一个仓库。

主站负责发现、进入、账户和状态；资料库、刷题站等成熟能力继续独立维护，通过统一设计与账户形成一个产品体系。

## 2. 当前产品地图

| 产品 | 当前地址 | 目标职责 | 当前决策 |
|---|---|---|---|
| HENU-Kit 主站 | Demo / 待接入 `henukit.cn` | 总入口、搜索发现、公告、账户入口 | 优先建设 |
| 资料库 | `study.superhuazai.me` | 资料浏览、检索、预览、下载 | 保持资料库定位，不增加刷题 |
| 刷题站 | `superhuazai.me/practice` | 练习、进度、错题、结果反馈 | 保留唯一刷题引擎 |
| 校园生活 | 待建设 | 美食榜单等轻量内容 | 统一基础完成后启动 |
| 统一账户 | 待建设 | 学生邮箱验证码登录、跨站会话 | 第一阶段核心基础设施 |

## 3. 阶段计划

### Phase 0：规则与盘点（第 1 周）

目标：团队对“做什么、怎么做、什么不做”达成一致。

- [x] 发布 Design System v1.0
- [ ] 确认正式域名与子域名结构
- [ ] 盘点主站、资料库、刷题站的技术栈与部署方式
- [ ] 列出可复用组件与重复能力
- [ ] 建立 Issue、PR、版本与发布约定
- [ ] 为各模块指定明确 Owner

退出条件：没有未解释的功能重叠；每项近期任务都有负责人和验收标准。

### Phase 1：主站 MVP（第 2–3 周）

目标：学生能从一个简洁入口找到并进入已有能力。

- [ ] 完成 `henukit.cn` 首页、工具导航和移动端适配
- [ ] 接入资料库与刷题站入口
- [ ] 增加工具状态、维护者和数据更新时间
- [ ] 固定展示“学生自主运营 · 非河南大学官方项目”
- [ ] 补齐 Loading / Empty / Error 等基础状态
- [ ] 建立基础访问统计，只采集必要数据

退出条件：新用户在 30 秒内能找到资料库或开始刷题；360px 宽度下关键流程可用。

### Phase 2：统一学生账户（第 3–5 周）

目标：一套学生邮箱账户连接主站、资料库和刷题站。

- [ ] 建设 `account.henukit.cn`
- [ ] 使用学生邮箱验证码完成注册/登录
- [ ] 使用 `notify.henukit.cn` 发送验证码
- [ ] MVP 采用阿里云 DirectMail 按量付费，并测试腾讯企业邮箱送达
- [ ] 实现验证码过期、单次使用、重发等待和邮箱/IP 限流
- [ ] 定义统一用户 ID、会话失效和退出登录流程
- [ ] 先接入主站，再接入刷题站，最后接入资料库

退出条件：三个产品识别同一个用户；敏感信息不进入日志或公开仓库；登录异常有明确恢复路径。

### Phase 3：统一体验（第 5–7 周）

目标：独立部署的产品在视觉和操作上属于同一个 HENU-Kit。

- [ ] 抽取颜色、排版、间距和基础组件 token
- [ ] 统一页头、账户入口、按钮、卡片、状态和页脚
- [ ] 资料库移除或拒绝新增重复刷题入口
- [ ] 资料库首页保留轻量“打开资料册”动效，移除重型滚动翻页与多层视差
- [ ] 资料库首页聚焦搜索、课程资料、我的下载和资料纠错/共建
- [ ] 刷题站保留专业练习体验，不复制资料归档
- [ ] 补齐移动端、键盘操作和对比度检查
- [ ] 评估将旧域名映射为 `study.henukit.cn`、`practice.henukit.cn`

退出条件：跨站切换时导航、身份和基础操作无明显割裂；不要求物理合并代码库。

### Phase 4：校园生活 MVP（第 8 周起）

目标：验证“美食榜单”是否值得成为长期模块。

- [ ] 先做浏览、筛选、详情和反馈，不做复杂社交
- [ ] 明确榜单数据来源、更新时间和排序规则
- [ ] 建立纠错与内容审核入口
- [ ] 用真实访问、收藏或反馈验证需求

退出条件：有明确使用数据和维护负责人，再决定是否扩展收藏、投稿或个性化。

## 4. 从旧 Roadmap 复用的内容

继续执行：

- 统一各项目 README 基本结构
- 补充截图、使用说明、维护状态和风险说明
- 整理 FAQ 和低门槛教程
- 抽取通用配置模板
- 建立清晰的社区贡献流程
- 各工具保留独立仓库，HENU-Kit 作为 umbrella repo 和总入口

暂缓：

- 适配其他学校
- 插件化平台
- 大而全的统一 Web 控制台
- 在需求未验证前整理大量校园系统接口

暂缓项不是否定，而是避免它们挤占统一入口、账户和现有产品整合的时间。

## 5. 不进入近期范围

- 在资料库重新实现刷题功能
- 将所有项目迁移到单体仓库
- 使用或改造河南大学校徽作为产品 Logo
- 冒充学校官方认证或官方平台
- 先做积分、社区、私信、推荐算法等复杂增长功能
- 依赖保存学生校园账号密码的登录方案

## 6. 发布门槛

每个阶段上线前必须满足：

- 有单一负责人、验收人和回滚方式
- 核心流程完成移动端人工测试
- 不泄露邮箱、验证码、Token、Cookie 或内部密钥
- 页面符合 Design System v1.0
- 明确非官方身份、数据来源和更新时间
- 错误状态能告诉用户下一步，而不是只显示失败

## 7. 成功指标

第一阶段只跟踪少量可解释指标：

- 首页到资料库/刷题站的成功点击率
- 学生邮箱验证码送达率和中位送达时间
- 登录完成率与验证码重发率
- 移动端核心流程成功率
- 每个模块的有效反馈数量与关闭时间

不以页面数量、功能数量或仓库数量衡量进展。
